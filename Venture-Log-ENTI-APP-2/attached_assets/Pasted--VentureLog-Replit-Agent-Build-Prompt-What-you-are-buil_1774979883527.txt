# VentureLog — Replit Agent Build Prompt

---

## What you are building

VentureLog is a focused SaaS web application for early-stage solo founders. It is built around one core loop: a founder writes a private journal entry, tracks a handful of business metrics, and asks an AI coach a question that is answered using the context of their own journal and data.

Do not add features that are not described here. Do not add user authentication or login until I explicitly ask for it in a follow-up prompt. Use a hardcoded single-user context for now so the full product can be built and tested without auth complexity.

Build the full application in phases in the order listed below. Complete each phase fully before moving to the next. After each phase, pause and show me what you have built so I can review it before continuing.

---

## Tech stack

Choose the stack you think is best for a modern SaaS web app. Suggestions if you need a starting point:

- **Frontend:** React with TypeScript
- **Styling:** Tailwind CSS with shadcn/ui components
- **Backend:** Node.js with Express or a framework of your choice
- **Database:** PostgreSQL
- **AI:** OpenAI API (I will provide my API key as an environment variable: `OPENAI_API_KEY`)

Do not implement Stripe, OAuth, or email sending in the initial build. Those come later.

## Application layout

The app has a persistent left sidebar and a main content area. The sidebar contains:

- App name "VentureLog" at the top
- Navigation links to: Home, Journal, Metrics, AI Coach, Decision Log, Investor Update
- At the bottom: a credit counter showing "X coach credits remaining"

The overall visual style is clean and professional. Use a blue-dominant palette:
- Primary: `#1F4E79`
- Action blue: `#2E75B6`
- Background canvas: `#F8FAFC`
- Cards and panels: `#FFFFFF`
- Body text: `#1A1A1A`
- Secondary text: `#64748B`

Typography: Inter for all UI text. JetBrains Mono for metric values and code blocks.

Border radius: 8px for cards and inputs, 12px for modals, 999px for badge pills.

Color is always a secondary indicator. Every color-coded state must also have a text label.

## Phase 1 — Founder journal (build this first)

### Data model

A `journal_entry` has:
- `id` (uuid)
- `content` (text, Markdown)
- `tags` (array of strings, chosen from fixed list: Product, Growth, Team, Fundraising, Operations, Finance, Reflection)
- `created_at` (timestamp)
- `updated_at` (timestamp)
- `is_promoted` (boolean, default false)

### Journal entry list screen

- Route: `/journal`
- Shows a chronological list of all journal entries, newest first
- Each entry card shows: first 2 lines of content, tags as colored badge pills, date, and an "Edit" button
- Top of the page has a "New Entry" button
- Left side has tag filter buttons (one per tag from the fixed list) and a search bar that does full-text search across entry content
- Empty state: "No entries yet. Start writing to capture your thinking."

### Journal entry editor screen

- Route: `/journal/new` and `/journal/:id/edit`
- A rich Markdown editor (use a library such as react-md-editor or similar)
- Tag selector below the editor — multi-select from the fixed tag list
- Auto-save to the server every 60 seconds when content has changed
- A save state indicator in the top-right corner of the editor: shows "Saved", "Saving…", "Unsaved changes", or "Save failed"
- A browser-local draft buffer (localStorage is fine for now) that saves content on every keystroke independently of server auto-save
- If the user tries to navigate away with unsaved changes or a "Save failed" state, show a warning modal: "Your last changes could not be saved to the server. A local draft has been preserved in this browser — reopen the editor to recover it." Two buttons: "Stay and retry" and "Leave (local draft preserved)"
- When the editor opens, check localStorage for a draft from a previous failed session. If one exists, show: "A local draft from [timestamp] was found. Restore it?" with "Restore draft" and "Discard draft" buttons
- A "Promote to Decision Log" toggle at the bottom of the editor. When toggled on, a small form expands below with fields: Decision title (required, max 120 chars), Context summary (required, max 500 chars), Options considered (at least one required), Chosen option (required). Submitting promotion saves the decision log item separately — the journal entry text is never copied to the decision log.

### Home dashboard screen

- Route: `/` (default landing page)
- Shows a daily journal prompt card at the top if no entry exists for today. Use this logic for the prompt:
  - If a metric has not been updated in more than 5 days: "Your [metric name] has not been updated since [date]. What drove that?"
  - If an open decision log item is more than 7 days old: "You logged a decision about [decision title] on [date] and left it open. Any update on the outcome?"
  - Otherwise: "What is the one thing from this week you want to make sure you remember?"
- Below the prompt: sparkline cards for the top 3 metrics (build this as a placeholder with dummy data for now — metrics are built in Phase 2)
- Below that: the 3 most recent shared decision log items

## Phase 2 — Metric dashboard (build this after Phase 1 is reviewed)

### Data model

A `metric` has:
- `id` (uuid)
- `name` (string)
- `class` (enum: revenue, retention, engagement, unit_economics)
- `period` (enum: daily, weekly, monthly)
- `direction` (enum: higher_is_better, lower_is_better, context_dependent)
- `formula_numerator` (string, nullable — for retention and unit_economics)
- `formula_denominator` (string, nullable — for retention and unit_economics)
- `is_archived` (boolean, default false)
- `created_at` (timestamp)

A `metric_value` has:
- `id` (uuid)
- `metric_id` (foreign key)
- `value` (decimal)
- `recorded_date` (date)
- `created_at` (timestamp)
- `is_correction` (boolean, default false)
- `original_value` (decimal, nullable — populated when is_correction is true)
- `correction_note` (string, nullable)

### Metric dashboard screen

- Route: `/metrics`
- Shows all active (non-archived) metrics as KPI cards in a responsive grid
- Each KPI card shows:
  - Metric name
  - Current value (JetBrains Mono font)
  - Previous period value
  - Absolute change and percentage change
  - A directional arrow (↑ or ↓) AND a text label ("Up 12%" or "Down 3%") — both always shown
  - Color: green background tint for favorable direction, red background tint for unfavorable. Direction is class-aware: for retention metrics with lower_is_better direction, an increase is red; for revenue metrics, an increase is green. Unit economics metrics with context_dependent direction show no color.
  - An inline "Log value" input that appears on hover or click
- A "+ Add metric" button that opens a modal with:
  - Metric class selector (4 options with brief descriptions)
  - Metric name field
  - Period selector
  - Direction selector (for retention and unit_economics, also show two formula input fields: "Numerator (e.g. churned customers)" and "Denominator (e.g. total customers at start of period)")
- Free plan limit: maximum 8 active metrics. If the user has 8 active metrics, the "+ Add metric" button is disabled with a tooltip: "Upgrade to Pro to add more metrics."
- Archived metrics are hidden from this view

### Metric detail screen

- Route: `/metrics/:id`
- Full time-series line chart showing up to 12 periods of history (use recharts or similar)
- Below the chart: a correction log table showing any corrections with original value, corrected value, corrector name, timestamp, and note
- "Log value" button that opens an inline form
- "Log correction" button that opens a form requiring the date, corrected value, and optional note — correction saves with `is_correction: true` and stores the original value
- A "Linked decisions" section listing any decision log items that are linked to this metric by metric ID

### Wire the home dashboard sparklines

Now that metrics exist, replace the placeholder sparklines on the home dashboard with real data from the top 3 most recently updated metrics.

## Phase 3 — AI coach (build this after Phase 2 is reviewed)

### Data model

A `coach_interaction` has:
- `id` (uuid)
- `task_type` (enum: trend_analysis, decision_review, weekly_retrospective)
- `context_snapshot` (jsonb — stores what data was used)
- `prompt` (text)
- `response` (text)
- `credits_used` (integer, default 1)
- `created_at` (timestamp)

A `journal_summary` has:
- `id` (uuid)
- `entry_id` (foreign key to journal_entry)
- `summary` (text)
- `created_at` (timestamp)
- `entry_updated_at` (timestamp — records the entry's updated_at at summary time, used to detect staleness)

### Background summarization

After every successful journal entry save, trigger a background job that calls the OpenAI API to generate a 2–3 sentence summary of the entry. Store the result in `journal_summary`. If the entry is edited and re-saved, the previous summary is invalidated and a new one is generated. If summarization fails, log the error silently — the entry remains fully saved and usable.

Use this system prompt for summarization:
```
You are summarizing a founder's private journal entry for use as AI coach context. 
Produce a 2-3 sentence factual summary of the key decisions, observations, blockers, 
and outcomes described. Do not add interpretation or evaluation. Do not reproduce 
sensitive personal content. Output only the summary with no preamble.
```

### AI coach panel

- Accessible from a persistent button in the sidebar (not a full-page route — opens as a right-side panel that overlays the current screen without navigating away)
- At the top of the panel: a task type selector with three clearly labeled options:
  1. **Analyse a metric trend** — "Interpret a specific metric's recent movement"
  2. **Review a decision I'm considering** — "Surface assumptions, risks, and relevant past decisions"
  3. **Generate my weekly retrospective** — "Summarise the past 7 days and suggest one focus for next week"
- Below the selector: a context preview that updates when the user selects a task type. Shows what data will be used (e.g. "I will use your MRR data from the last 3 months and any linked decision log items")
- A free-text input for the user's question
- A "Submit" button that shows credit cost: "Submit (uses 1 credit)"
- Credit counter showing remaining credits (hardcode to 15 for now — auth and billing come later)
- Response area that streams the AI response token by token

### Context assembly per task type

**Trend analysis:**
- Identify which metric the user is asking about (use keyword matching against metric names)
- Fetch the last 3 period values for that metric
- Fetch decision log items linked to that metric by metric ID
- Context snapshot stores: metric name, last 3 values with dates, linked decision titles

**Decision review:**
- Fetch the last 14 days of journal summaries (summarized entries only; note how many entries in that period are unsummarized)
- Fetch a snapshot of all current metric values (active metrics only)
- Fetch titles and chosen options of all closed decision log items
- Context snapshot stores: summary count, unsummarized count, metric snapshot, past decision titles

**Weekly retrospective:**
- Fetch journal summaries from the last 7 days (note summarized count vs total entry count in that period)
- Fetch metric value changes in the last 7 days for active metrics only
- Context snapshot stores: summarized entry count, total entry count, metric changes

### AI response generation

Use the OpenAI API with the following approach for all three task types:

System prompt:
```
You are a focused AI coach for early-stage startup founders. You are constrained to 
three task types: trend analysis, decision review, and weekly retrospective. You must 
always begin your response with a context statement in exactly this format:
"Context used: [N of M] journal entries summarized from [date range]; [metric] data 
for [N periods]; [N] past decisions referenced."

Then, if context is sparse (fewer than 3 journal summaries for the relevant period, 
or missing metric data), add: "My context for this period is limited — treat this as 
a starting point rather than a grounded analysis."

For unit economics metrics, always add: "Unit economics metrics involve user-defined 
formulas and context-dependent direction. My interpretation may not reflect your 
specific business model."

Confidence rules by claim type:
- High confidence (revenue, retention with full context): all claim types permitted
- Medium confidence (engagement, sparse context, decision review, retrospective): 
  observations, hypotheses, questions, and conditional statements only. No causal 
  claims ("X caused Y"), no predictions stated as certainties, no prescriptive claims 
  stated without qualification.
- Low confidence (unit economics, fewer than 3 summaries, missing metric data): 
  observations, explicit data gap acknowledgements, and investigative questions only.

You must never use the phrases "clearly", "definitely", or "the data shows" when 
operating at medium or low confidence. You must never make causal claims about unit 
economics metrics.

If a question does not fit one of the three task types, respond only with:
"This question doesn't match my three supported task types. I can help with:
1. Trend analysis — interpreting a specific metric's recent movement
2. Decision review — evaluating a decision you're considering
3. Weekly retrospective — summarising your past 7 days
Please rephrase your question or select a task type from the panel."
```

### Challenge mode

After any AI response, show a "Challenge this" button below the response. Clicking it calls the API again with the original context plus a request for a steelman counterargument. The challenge response inherits the same confidence level and opens with the same context statement. Costs 1 additional credit.

## Phase 4 — Shared decision log (build this after Phase 3 is reviewed)

### Data model

A `decision_log_item` has:
- `id` (uuid)
- `source_entry_id` (foreign key to journal_entry, nullable — null if created directly)
- `linked_metric_id` (foreign key to metric, nullable)
- `title` (string, max 120 chars)
- `context_summary` (text, max 500 chars)
- `options_considered` (jsonb array of strings)
- `chosen_option` (string)
- `expected_outcome` (string, nullable)
- `actual_outcome` (string, nullable)
- `lessons_learned` (string, nullable)
- `tags` (array of strings from fixed list)
- `status` (enum: open, closed)
- `has_comments` (boolean, default false)
- `is_archived` (boolean, default false)
- `created_at` (timestamp)

A `decision_comment` has:
- `id` (uuid)
- `decision_id` (foreign key)
- `author_name` (string — hardcoded for now, will be real user later)
- `content` (text)
- `created_at` (timestamp)

### Decision log screen

- Route: `/decisions`
- Table view of all non-archived decision log items
- Columns: title, tags, status badge (Open/Closed), linked metric name (or "—"), date, actions
- Filter bar: status filter (All / Open / Closed), tag filter, search by title
- "New Decision" button that opens the creation form directly (not requiring a journal entry)
- Click a row to open the decision detail view

### Decision detail panel/modal

- Shows all fields
- "Close decision" button (when status is Open) — opens a form requiring actual outcome text, optional lessons-learned note. No numeric score field.
- Comment thread below the decision fields
- "Add comment" input at the bottom
- Delete button: visible only if `has_comments` is false. Shows "Delete" label.
- Archive button: visible always. Shows "Archive" label. When `has_comments` is true, "Delete" is replaced entirely by "Archive".
- Archived items: hidden from the main decision log view. A toggle "Show archived" at the top reveals them with a muted "Archived" badge.

## Phase 5 — Investor update generator (build this after Phase 4 is reviewed)

### Data model

A `saved_draft` has:
- `id` (uuid)
- `content` (jsonb — stores the four sections: highlights, metrics, ask, low_points)
- `date_range_start` (date)
- `date_range_end` (date)
- `created_at` (timestamp)
- `updated_at` (timestamp)

Only one saved draft exists at a time (upsert on save).

### Investor update screen

- Route: `/investor-update`
- A date range selector at the top (default: last 30 days)
- A "Generate draft" button
- On generation, assemble the draft from:
  - **Highlights** and **Low points**: journal entries in the date range tagged with "Fundraising" or "Growth". Show the entry tag, date, and first 100 characters of content as the draft seed. (For now, include all such entries — a "mark as shareable" flag comes with the auth/permissions layer later.)
  - **Metrics**: for each active metric, show name and the value change over the date range (current value vs value at start of date range, or earliest available value if start-of-range value is missing)
  - **Ask**: blank text field
- The draft renders in four editable rich-text sections
- A "Copy to clipboard" button that copies the full formatted draft as plain text
- A persistent note below the button: "Paste this into your email client to send. Direct email sending will be available in a future update."
- Auto-save the draft to the database after any edit (debounced, 3 seconds)
- On page load, fetch the saved draft and populate the editor
- If a saved draft exists and the user clicks "Generate draft" again, show a confirmation modal: "This will replace your saved draft from [date]. Continue?" with "Replace" and "Cancel" buttons.
- Empty state for Highlights/Low points when no qualifying journal entries exist: "No entries tagged Fundraising or Growth were found for this period. Tag relevant journal entries to populate these sections."

## General build rules

1. Build a proper REST API backend — do not use client-side only storage for anything except the local draft buffer.
2. Use database migrations. Every schema change should be a migration file.
3. Handle errors gracefully. Every API call in the frontend should have a loading state and an error state.
4. Mobile-responsive layout. The app should be usable on a phone browser without horizontal scrolling.
5. Do not add features I have not asked for. If you think something is missing, ask me before building it.
6. After each phase is complete, write a brief summary of what was built, what environment variables are needed, and what I should test first.
